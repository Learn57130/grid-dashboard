grid_strike guide
hyperliquid ◀ builder

A grid is a ladder of buy orders across a price range. Each rung buys when price dips to it and sells a little higher — harvesting the up-and-down chop. It profits while price stays inside the range; its risk is price trending out, leaving you holding a losing position. This page explains every knob and how to set it.

the mental model read this first

You never set "number of levels" or "spacing" directly — they are derived from the other params:

# how many rungs, and how far apart
levels  = total_amount ÷ min_order  (capital limit — usually what binds)
spacing = range ÷ (levels − 1)   where range = (end − start) / start

Example: $50 ÷ $10 = 5 levels; a 17.5% range ÷ 4 gaps = 4.4% spacing. Want more, tighter rungs? Raise total_amount or lower min_order. Want wider rungs? Widen the range.

range where the grid sits

start_price low
Bottom rung of the grid. Set it below the current price.tune: drag the line on the chart, or use the ±% buttons around a center price.
end_price high
Top rung. Must be above start_price. Wider gap = bigger range = wider spacing for the same number of levels.tune: bracket where you expect price to oscillate. Too wide and rungs sit too far apart to fill.
limit_price stop
Invalidation. Sits below start_price. If price breaks below it, the grid halts — your downside cap.tune: set it at the price where your thesis is wrong. Draggable on the chart.

sizing how much, how many rungs

total_amount_quote USDC
Total capital the grid deploys, split across rungs.tune: raise it for more levels / a tighter grid. At 1× leverage this is also the margin used.
min_order_amount_quote USDC
Notional per rung. Hyperliquid's minimum is ~$10.tune: lower it for more levels from the same capital (down to ~$10).
min_spread_between_orders frac
Floor on spacing (0.001 = 0.1%). Rungs can't get closer than this.tune: rarely the binding limit — leave at 0.001 unless you pack lots of capital into a tight range.

take profit profit per rung

take_profit %
How far above its buy each rung sells. Linked to spacing by default (⛓) so each rung sells at the next rung up — a coherent grid.tune: leave linked for a real grid. Unlink only for fixed small-TP "rung scalping".
The coherence rule. For a real grid, take_profit ≈ spacing. If TP is far below spacing (e.g. 0.15% vs 4.4%), you're not gridding — you're running a few disconnected scalp points and the wide spacing is cosmetic. The ⛓ link keeps them equal so the levels actually matter.

execution how orders reach the book

max_open_orders
How many rungs rest in the book at once. With the default 2, only the 2 rungs nearest price are live — a sliding window that follows the market. Set it to your level count to arm the full ladder.tune: = level count to blanket the range (catches fast moves, ties up more margin); low (2) for a lean price-following pair.
max_orders_per_batch
Orders placed per round. Nearest rungs first.tune: raise to arm many rungs faster; 1 is gentle on the exchange.
order_frequency s
Minimum seconds between placement rounds. Only matters while filling empty slots (startup, or after fills) — does nothing once the ladder is full.tune: 3s default is deliberately gentle. Lower to arm faster, at the cost of more frequent API bursts.
activation_bounds blank=null
Only place orders within this fraction of mid-price. Blank = always active across the whole range.
keep_position
On stop, keep the position you accumulated instead of closing it out.

market & account

connector_name
hyperliquid_perpetual = USDC-margined perps; hyperliquid = spot. (BTC spot on HL is illiquid — use the perp.)
trading_pair
Perps use BASE-USD (e.g. BTC-USD); spot uses BASE-USDC.
side
BUY = long grid (accumulates as price falls). SELL = short grid.
leverage x · perp
1× = no leverage (safest, near-spot). Higher adds liquidation risk. Ignored on spot.tune: keep 1× for study. Liquidation is remote at 1×.
position_mode
Hyperliquid supports ONEWAY only — HEDGE errors on start.
id
Config identifier — also the bot instance name once deployed.

economics assumptions estimate only — not written to config

These feed the live per-cycle math on the builder. They shape the estimate, not the deployed bot.

# what one buy→sell cycle nets
cost/cycle = 2 × maker_fee + funding × hold_hrs  (≈ 0.03% fee + carry)
net/cycle  = take_profit − cost/cycle
maker_fee %
Your maker fee. HL perp base ≈ 0.015%. Both grid legs are LIMIT_MAKER, so a round trip costs ~2×.
funding_hr %/hr
Hourly funding (perp carry). Auto-filled live; type to override. Positive = longs pay.
hold_hrs
Average hours a rung's position stays open — scales the funding cost per cycle.
balance USDC
Account equity, for the margin-use estimate. "use as balance" fills it from a fetched wallet.

risk what can go wrong

A grid earns many small wins and risks one big loss. The wins are the take_profit cycles; the loss is the inventory you accumulate if price trends out of the range — you keep buying into the fall down to the limit. The builder's drawdown chart shows that worst case. On a perp, funding also bleeds a position that sits open too long — a thin TP can go negative if a rung is held for days.

safety before real money

Use an api_wallet key, never your main private key. The agent/API wallet from app.hyperliquid.xyz/API can trade but cannot withdraw — so a leaked key can't drain funds. The builder only ever needs your public address for read-only balance.
  1. Keep leverage 1× — liquidation is remote.
  2. Fund the perp account: transfer USDC spot → perp in the HL app (BTC lives on the perp).
  3. Watch a grid succeed and fail for free first — paper trade on a supported venue, or Hyperliquid testnet.
  4. Size it as tuition you can fully lose, not capital that matters.
